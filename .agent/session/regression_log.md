## #2 [2026-04-12] テキスト変位式と二重変換の修正 — Session 9d60fe32

### 試行した修正
- CTM（座標変換行列）が SDK と Renderer で二重適用されている問題を発見し修正
- `compute_offset` のバイト単位演算を整数演算に変更
- グリフメトリクスロジックのリグレッションを修復

### 引き起こしたリグレッション
- Renderer で `view_transform` のみ使用するようにしたことで、FontMatrix 継承が壊れた
- Type 0 フォントで 1000x スケールエラーが発生

### 学んだ不変条件
- SDK が生成するグリフパスには CTM が焼き込まれている前提で Renderer を設計してはならない
- `compute_offset` はバイト列を単一 u32 整数として扱う必要がある

## #3 [2026-04-12] 座標系同期と FontMatrix 継承 — Session e8855a35

### 試行した修正
- Renderer の CTM に Y 軸反転を反映させ、SDK 側でグリフを `-fs` で抽出するよう変更
- FontMatrix のデフォルトを `[0.001, 0, 0, 0.001, 0, 0]` に統一
- Type 0 フォントが子 CIDFont の FontMatrix を継承するよう修正

### 引き起こしたリグレッション
- FontMatrix と font_size (Tfs) のスケーリング順序が不明確になった
- SDK と Renderer のスケール分担の境界が曖昧なまま

### 累積的不変条件 (Cumulative Invariants)

- **座標系**: PDF 空間 (Bottom-Up) と Vello 空間 (Top-Down) の変換は、必ず Document-level CTM を通さなければならない。
- **スケーリング**: `FontMatrix` (0.001) のスケーリングは、PDF の `W` 配列適用**後**に行わなければならない。
- **検証 (NEW)**: 自動テストによる定量的証拠がない「完了報告」は、将来のリグレッションを隠蔽するリスクがある。修正は必ずテストを伴わなければならない。

### 学んだ不変条件
- FontMatrix は Type 0 → CIDFont の親子関係で継承する
- スケーリングチェーン全体（FontMatrix → Tfs → CTM）の責務分担を文書化しないと、修正が毎回異なる仮定に基づいてしまう

## 累積的な教訓 (Invariants Summary)

> 以下は全エントリから抽出された不変条件の要約。ログが肥大化しても、これは保持すること。

1. CMap 修正だけではテキスト描画は直らない（変位計算も関与する）
2. CTM の適用は SDK か Renderer のどちらか一方のみで行う
3. `compute_offset` はバイト列を単一 u32 として処理する
4. FontMatrix は Type 0 → CIDFont で継承する
5. スケーリングチェーン（FontMatrix → Tfs → CTM）の責務分担を文書化する必要がある
6. W 配列 vs フォントメトリクスの優先ポリシーが必要
7. **テスト無しの「完了」は信用できない**
